## 1. このファイルの役割

`DESIGN.md`は「何を、なぜ、どのように作るか」を定義する。`BACKLOG.md`は、その設計を実装可能な小さな作業へ分解し、次に何へ着手できるかを示す。

本ファイルは、Backlog.mdの次の考え方を取り入れた単一ファイル版として運用する。

- 1タスクを1回の作業・レビューで完了できる大きさにする。
- 実装前に説明と受け入れ条件をレビューする。
- 実装直前に現在のコードを調査し、実装計画を追記する。
- 実装後にテスト、差分、受け入れ条件をレビューする。
- 優先度、依存関係、完了条件を構造化する。
- 完了した作業もGit履歴として残す。

現時点では公式Backlog.md CLIの`backlog/`ディレクトリは導入しない。タスクが30件を超える、複数人・複数エージェントで並行作業する、Kanban表示が必要になる、のいずれかが発生した場合に移行を検討する。

## 2. 運用ルール

### 2.1 ステータス

| ステータス | 意味 |
| --- | --- |
| `Todo` | 依存関係が満たされれば着手できる |
| `In Progress` | 現在作業中。原則1件だけにする |
| `Blocked` | 外部条件または別タスク待ち |
| `Done` | 受け入れ条件とDefinition of Doneを満たした |
| `Deferred` | MVP対象外。着手条件が成立するまで保留 |

### 2.2 優先度

| 優先度 | 意味 |
| --- | --- |
| `P0` | MVP成立に必須。上から順に進める |
| `P1` | MVPの安全性・使いやすさに必要 |
| `P2` | MVP完成後の改善 |

### 2.3 更新方法

1. 着手前に依存タスクが`Done`であることを確認する。
2. 対象タスクだけを`In Progress`へ変更する。
3. 現在のコードを調査し、「実装計画」へ変更対象と検証手順を追記する。
4. 計画を確認してから実装する。
5. 受け入れ条件を1項目ずつ検証し、チェックを付ける。
6. Definition of Doneを満たしたら`Done`へ変更する。
7. 新しい要求は既存タスクへ無理に追加せず、新しいIDで追加する。

## 3. Definition of Ready

次の条件を満たすタスクだけ着手できる。

- 目的と期待する成果が明確である。
- 依存タスクがすべて`Done`である。
- 受け入れ条件が観測可能な表現になっている。
- MVPスコープ内か、例外として着手する理由が明記されている。
- 外部サービスの権限や人間の判断が必要な場合、その作業が明記されている。

## 4. Definition of Done

すべてのタスクで次を満たす。

- [ ] 受け入れ条件をすべて満たした。
- [ ] 関連する自動テストまたは再現可能な手動検証を実行した。
- [ ] 型チェック、テスト、ビルドなど既存の品質確認が通った。
- [ ] 秘密情報、個人情報、不要な生成物をコミット対象へ加えていない。
- [ ] `DESIGN.md`と実装が矛盾する場合、設計判断を先に更新した。
- [ ] `README.md`の利用手順へ影響する場合、同じタスクで更新した。
- [ ] 実装結果と検証結果をタスクの「完了記録」へ追記した。

## 8. 調査から採用したポイント

Backlog.mdは、Markdownを正本としてAIエージェントと人間が共同で作業を管理するツールで、2026年7月時点では次を重視している。

- 説明、受け入れ条件、Definition of Doneを持つ小さなタスク
- 1タスクを1コンテキスト・1レビュー単位にする運用
- 仕様、計画、コードの3段階で人間が確認するチェックポイント
- マイルストーンと依存関係による実行順の可視化
- ローカルファーストかつGitで追跡できるMarkdown

GitHub Issuesの公式ガイドでも、作業の細分化、依存関係、優先度などの構造化メタデータが推奨されている。本ファイルでは、これらを外部サービスなしで扱える最小構成にした。
